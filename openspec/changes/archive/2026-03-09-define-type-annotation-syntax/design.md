## Context

TypingLua 项目已集成 Lua 5.4.7 源码并能成功编译。现在需要为 Lua 定义一套类型注解语法，作为后续实现类型检查器的基础。

Lua 的注释机制：
- 短注释：`--` 开头，到行尾结束（`llex.c` 第 457-474 行）
- 长注释：`--[[` 开头，`]]` 结束
- Lexer 处理时会完全跳过注释内容，不保留任何语义信息

业界已有方案：
- **EmmyLua/LuaLS**：使用 `---@` 前缀的注释注解，社区已广泛采用
- **Teal**：修改语法，使用内联 `: type` 标注，不兼容标准 Lua
- **Python type hints**：PEP 484，使用 `: type` 内联标注 + `-> type` 返回值标注

## Goals / Non-Goals

**Goals:**
- 定义一套完整的类型注解语法规范，覆盖基础类型、复合类型和常用注解指令
- 语法基于注释（`---@`），确保注解后的代码仍是合法的标准 Lua
- 语法设计与 EmmyLua/LuaLS 风格保持高度兼容，降低社区迁移成本
- 提供 EBNF 形式的语法描述，为后续解析器实现提供精确规范
- 提供足够的示例覆盖常见使用场景

**Non-Goals:**
- 不实现类型注解的解析器（后续 change 处理）
- 不实现类型检查逻辑
- 不修改 Lua 源码
- 不定义泛型语法（后续扩展）
- 不定义类/接口/枚举等高级类型结构（后续扩展）

## Decisions

### Decision 1: 采用注释风格（`---@`）而非内联风格（`: type`）

**选择：** 使用三短线注释前缀 `---@` 作为类型注解的标记。

**理由：**
- 保持与标准 Lua 的完全兼容——带注解的代码可直接被原版 Lua 解释器执行
- 与 EmmyLua/LuaLS 生态系统兼容，社区用户无需重新学习
- 不需要修改 Lua 的核心语法（lexer/parser），注解作为独立层解析

**放弃的替代方案：**
- 内联 `: type` 风格（如 Teal）——需要修改 Lua 语法，破坏兼容性
- 使用 `--:` 或其他自定义前缀——与现有工具链不兼容，社区认知成本高

### Decision 2: 基础类型集合

**选择：** 定义以下基础类型关键字：

| 类型 | 含义 | 对应 Lua type() |
|------|------|-----------------|
| `nil` | 空值 | `"nil"` |
| `boolean` | 布尔值 | `"boolean"` |
| `number` | 数值（含整数和浮点） | `"number"` |
| `integer` | 整数子类型 | `"number"` (Lua 5.4 整数) |
| `string` | 字符串 | `"string"` |
| `table` | 表（无具体结构） | `"table"` |
| `function` | 函数（无具体签名） | `"function"` |
| `thread` | 协程 | `"thread"` |
| `userdata` | 用户数据 | `"userdata"` |
| `any` | 任意类型（跳过检查） | — |
| `void` | 无返回值 | — |

**理由：** 前 7 个直接对应 Lua 的 `type()` 返回值，`integer` 是 Lua 5.4 新增的子类型，`any` 和 `void` 是类型系统的常规需求。

### Decision 3: 三种核心注解指令

**选择：** 首先支持三种注解指令：

```lua
---@type <type-expr>          -- 变量类型
---@param <name> <type-expr>  -- 函数参数类型
---@return <type-expr>        -- 函数返回值类型
```

**理由：** 这三种覆盖了最高频的使用场景（变量声明、函数签名），是类型系统的最小可用集。

### Decision 4: 复合类型语法

**选择：**

| 语法 | 含义 | 示例 |
|------|------|------|
| `T1\|T2` | 联合类型 | `number\|string` |
| `T?` | 可选类型（等价于 `T\|nil`） | `string?` |
| `T[]` | 数组类型（等价于 `table<integer, T>`） | `number[]` |
| `table<K, V>` | 键值表类型 | `table<string, number>` |
| `fun(params): ret` | 函数签名类型 | `fun(x: number): string` |

**理由：** 这些是最常用的复合类型构造，覆盖了绝大多数实际代码中的类型表达需求。语法与 LuaLS 保持一致。

### Decision 5: 语法规范使用 EBNF 格式

**选择：** 用 EBNF（Extended Backus-Naur Form）来形式化描述类型注解的语法。

**理由：** EBNF 是描述编程语言语法的标准方式，便于后续实现解析器时直接对照。Lua 官方手册也使用类似格式描述语法。

## Risks / Trade-offs

### [与 LuaLS 不完全兼容] → 明确差异文档
LuaLS 的注解语法非常丰富（包含 class、alias、overload 等），我们首期只覆盖基础子集。需要在文档中明确说明当前支持范围和后续扩展计划。

### [语法扩展性] → 预留扩展点
首期语法设计需要为后续扩展（泛型、类、接口）预留空间，避免语法冲突。`---@` 前缀 + 指令名的结构天然具备扩展性——新增指令不影响已有语法。

### [注释解析复杂度] → 独立解析层
类型注解嵌套在 Lua 注释中，需要在 Lua lexer 之外实现独立的注解解析器。这增加了系统复杂度，但保持了与标准 Lua 的隔离。

## Open Questions

- 是否需要支持多行注解（一个 `---@param` 跨多行描述）？
- 字面量类型（如 `"hello"` 或 `42` 作为类型）是否纳入首期？
- 元组返回值（`---@return number, string`）的语法如何处理——逗号分隔还是括号包裹？
